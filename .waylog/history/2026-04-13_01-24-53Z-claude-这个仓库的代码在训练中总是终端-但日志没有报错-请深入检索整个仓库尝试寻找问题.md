## 根本原因分析

### 问题 1 — `height_scanner` 占用 GPU 但从未使用（直接内存浪费）

[velocity_env_cfg.py:68](source/unitree_rl_lab/unitree_rl_lab/tasks/locomotion/robots/g1/29dof/velocity_env_cfg.py#L68) 里创建了 height_scanner：
```python
height_scanner = RayCasterCfg(
    pattern_cfg=patterns.GridPatternCfg(resolution=0.1, size=[1.6, 1.0]),  # 160条射线
    ...
)
```

[velocity_env_cfg.py:387](source/unitree_rl_lab/unitree_rl_lab/tasks/locomotion/robots/g1/29dof/velocity_env_cfg.py#L387) 里设置了它的更新频率：
```python
self.scene.height_scanner.update_period = self.decimation * self.sim.dt  # 仍然每步计算！
```

但在 ObservationsCfg 里观测是**注释掉的**：
```python
# height_scanner = ObsTerm(func=mdp.height_scan, ...)   ← 从未使用
```

**结果**：每一步为 4096 个环境各发射 160 条射线（共 **655,360 条/步**），纯粹浪费 GPU 内存和计算资源，且这个压力在训练进行中一直持续。

---

### 问题 2 — `gpu_max_rigid_patch_count` 可能在运行中被突破

[velocity_env_cfg.py:382](source/unitree_rl_lab/unitree_rl_lab/tasks/locomotion/robots/g1/29dof/velocity_env_cfg.py#L382)：
```python
self.sim.physx.gpu_max_rigid_patch_count = 10 * 2**15  # = 327,680
```

G1 29 自由度人形机器人有 30+ 个刚体链接，4096 个环境同时运行。当机器人在训练中倒地或多个部位同时碰撞时，总接触 patch 数可能超出上限。PhysX 超出限制后会**静默终止**，不产生 Python traceback。这就是为什么每次停止的时间点不同——取决于随机初始化和训练进度。

---

### 问题 3 — 默认使用 `wandb` logger，后台线程可能静默崩溃

[cli_args.py:51](scripts/rsl_rl/cli_args.py#L51)：
```python
default="wandb"
```

wandb 使用独立后台进程同步数据。若服务器网络连接在 20~600 次迭代后超时或断开，wandb 的后台进程会调用 `os._exit()` 终止主进程，不产生 Python 异常。
