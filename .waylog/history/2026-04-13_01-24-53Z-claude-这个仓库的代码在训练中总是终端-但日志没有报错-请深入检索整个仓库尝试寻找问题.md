### 问题 2 — `gpu_max_rigid_patch_count` 可能在运行中被突破

[velocity_env_cfg.py:382](source/unitree_rl_lab/unitree_rl_lab/tasks/locomotion/robots/g1/29dof/velocity_env_cfg.py#L382)：
```python
self.sim.physx.gpu_max_rigid_patch_count = 10 * 2**15  # = 327,680
```

G1 29 自由度人形机器人有 30+ 个刚体链接，4096 个环境同时运行。当机器人在训练中倒地或多个部位同时碰撞时，总接触 patch 数可能超出上限。PhysX 超出限制后会**静默终止**，不产生 Python traceback。这就是为什么每次停止的时间点不同——取决于随机初始化和训练进度。

## 根本原因：`std` 参数没有正值约束

看 traceback 链路：
```
actor_critic.py line 119: self.distribution.sample()
    → Normal(loc=mean, scale=self.std)  ← std 是直接的 nn.Parameter，无正值约束
    → torch.normal(..., self.scale)     ← scale < 0 → 报错
```

在 rsl_rl 的 `ActorCritic` 里，`std` 是一个原始的 `nn.Parameter`，**没有通过 `exp()` 或 `softplus()` 保证正值**。经过 26559 轮训练，梯度更新把某个维度的 `std` 推过了 0。

这是一个**数值稳定性问题**，触发时间不固定，取决于训练动态。根本诱因通常是：

1. **观测未归一化** — 当前配置 `empirical_normalization = False`，随着训练深入，网络输入尺度变化，梯度逐渐不稳定
2. **entropy_coef 过小** — `entropy_coef=0.01` 的熵正则化最终不足以阻止 `std` 衰减到 0
