## 根本原因：`std` 参数没有正值约束

看 traceback 链路：
```
actor_critic.py line 119: self.distribution.sample()
    → Normal(loc=mean, scale=self.std)  ← std 是直接的 nn.Parameter，无正值约束
    → torch.normal(..., self.scale)     ← scale < 0 → 报错
```

在 rsl_rl 的 `ActorCritic` 里，`std` 是一个原始的 `nn.Parameter`，**没有通过 `exp()` 或 `softplus()` 保证正值**。经过 26559 轮训练，梯度更新把某个维度的 `std` 推过了 0。

这是一个**数值稳定性问题**，触发时间不固定，取决于训练动态。根本诱因通常是：

1. **观测未归一化** — 当前配置 `empirical_normalization = False`，随着训练深入，网络输入尺度变化，梯度逐渐不稳定
2. **entropy_coef 过小** — `entropy_coef=0.01` 的熵正则化最终不足以阻止 `std` 衰减到 0

## 结论：`empirical_normalization = True` 与 RoboJudo 完全兼容

### 关键逻辑链

**训练 → 导出的流程**（[play.py:166-176](scripts/rsl_rl/play.py#L166-L176)）：

```python
# 1. 提取归一化层
if hasattr(policy_nn, "actor_obs_normalizer"):
    normalizer = policy_nn.actor_obs_normalizer   # ← 当 empirical_normalization=True 时存在
else:
    normalizer = None

# 2. 导出时将归一化层打包进 JIT 模型
export_policy_as_jit(policy_nn, normalizer=normalizer, ...)
```

`export_policy_as_jit`（来自 `isaaclab_rl`）在 `normalizer` 不为 None 时，会创建一个包装模块：
```
JIT 模型 forward(obs) = actor( normalizer(obs) )
```
归一化层被**烘焙进** `.pt` 文件，成为推理链路的一部分。

**RoboJudo 的推理**（[base_policy.py:86-88](C:/ArtiIntComVis/RoboJuDo/robojudo/policy/base_policy.py#L86-L88)）：
```python
obs_tensor = torch.from_numpy(obs).unsqueeze(0).float().to(self.device)
actions_tensor = self.model(obs_tensor)   # ← 归一化在 JIT 内部自动执行
```

RoboJudo 直接调用 JIT 模型，**不需要也不应该**手动做归一化——因为归一化已经在模型内部了。

---

### 总结

| 配置 | 导出的 JIT 模型包含什么 | RoboJudo 行为 |
|------|----------------------|--------------|
| `empirical_normalization = False`（旧） | 仅 actor 网络 | 直接推理，无归一化 |
| `empirical_normalization = True`（新） | normalizer + actor 网络 | 推理时内部自动归一化 |

两种情况 RoboJudo **都不需要做任何修改**，归一化与否对 RoboJudo 完全透明。

## 分析结论：这个问题在当前设计中**潜在存在**，扩大转身范围后会明显暴露

---

### 问题机制

转身时有两种路径可以产生角速度：

| 路径 | 关节 | 当前配置 | 结果 |
|------|------|---------|------|
| **A — 自然方式**：髋部 yaw 偏转 → 脚步外拨 → 踏步转身 | `hip_yaw_joint` | 被 `-1.0` 强惩罚 | **被压制** |
| **B — 异常方式**：腰部 yaw 旋转 → 上身扭转 → 等到失衡再迈步 | `waist_yaw_joint` | 被 `-1.0` 强惩罚 | 代价也高，但小范围内仍可用 |

当前因 `ang_vel_z=±0.2 rad/s` 极小，路径 B 的微小腰部扭转就足以追踪指令，问题被掩盖。**一旦按之前建议扩大转身范围至 `±1.0 rad/s`**：

- 微小腰部扭转不够用了，策略被迫加大腰部旋转
- `hip_yaw` 仍受惩罚，脚步无法主动外拨
- 结果：**上身扭到极限 → 重心失稳 → 脚步被动追赶** ← 正是师弟遇到的问题

---

### 根本原因

**`.*_hip_yaw_joint` 被纳入 `joint_deviation_legs` 惩罚，是"上身先转"问题的直接诱因。**

`hip_yaw_joint` 是脚步主动外拨转向的核心关节。惩罚它等于强制策略放弃自然的踏步转身方式，只剩下扭上身这一条路。
